# D3 Semantic Split Prompt

> **用途**: 规则无法准确切分长段落时,调用 Claude 按语义切分
> **调用点**: `agent_d3/ai_helpers.py::semantic_split()`
> **v0.1 可选**: 如果时间紧,先用简单字数切分,v0.5 再上

---

```text
将以下长段落拆成多个短段,每段不超过 {max_words} 词.

## 要求

1. **按语义边界切分**
   - 在论点切换处、场景切换处、时间跳跃处切分
   - 不在句子中间硬切

2. **每段独立可读**
   - 拆分后每段应该独立成立
   - 读第二段不需要回看第一段就能理解

3. **保留所有原始信息**
   - 不删减内容
   - 不增加内容
   - 不改写语言风格

4. **目标段数**
   - 原段字数 / max_words 向上取整
   - 例如: 原 150 词 / max 50 词 = 至少 3 段

## 原段落

```
{paragraph}
```

## 输出格式

直接输出拆分后的段落,段落之间用**一个空行**分隔.

不要:
- 编号 (不要 1. 2. 3.)
- 小标题
- 任何前言或解释

## 示例

**输入** (max_words=50):

"2024 年 AutoGPT 刷屏那阵,很多人以为 Agent 的时代来了.实际上 AutoGPT 在真实任务上
跑不过 3 轮就崩——不是因为 LLM 不够强,是因为它没有状态机.到 2025 年下半年,主流框
架才开始补这个缺失,而这个时候 Claude Code 已经在用完全不同的思路:把状态机放在工
具里,而不是在 Agent 里."

**输出**:

2024 年 AutoGPT 刷屏那阵,很多人以为 Agent 的时代来了.实际上 AutoGPT 在真实任务上跑不过 3 轮就崩——不是因为 LLM 不够强,是因为它没有状态机.

到 2025 年下半年,主流框架才开始补这个缺失,而这个时候 Claude Code 已经在用完全不同的思路:把状态机放在工具里,而不是在 Agent 里.

## 边界情况

- 如果原段本来就短 (≤ max_words),直接返回原段不变
- 如果原段是代码块或表格,不要切分,返回原段
- 如果原段含 Markdown 引用 (>),保留 > 前缀
```
