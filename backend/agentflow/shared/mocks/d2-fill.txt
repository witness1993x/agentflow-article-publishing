我最初以为 subagent 就是线程池的 LLM 版本:一个主 agent 把任务分发给几个子 agent,子 agent 并发跑,结果汇总。3 月底我照这个直觉设计了一个 6 节点的 agent,跑真实任务时端到端比单 agent 还慢 30%。这个反直觉的数字让我第一次停下来想:问题不在我代码写错了,问题在我对 subagent 的心智模型本身就不对。

[IMAGE: subagent 并行执行示意图]

真正的代价不是计算,是上下文切换。每次一个 subagent 接过任务,它需要重新建立一部分上下文——哪怕主 agent 已经做过类似的推理。这份"重入成本"在单次调用里看不出来,跑 20 次才开始显形。换句话说,subagent 不是免费的加速,它是一种有代价的抽象,代价正好藏在你以为最省时间的地方。

后来我换了个思路:不把 subagent 当成"让 LLM 并行",而是当成"让推理单位变小"。一旦视角切到这里,很多设计决策突然变清晰了——哪些任务适合拆,哪些任务死活不该拆,判断依据从"能不能并发"变成"这块推理是不是独立的意图"。这个转变花了我大概两周才内化,但它改写了我后面所有 agent 架构的起点。
