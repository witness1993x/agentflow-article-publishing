# Social Content Review Skill — 结构建议

> **场景定位**：自动化社媒运营全链路审核（话题发现 → 内容生成 → 图片生成 → 多渠道发布）
> **覆盖渠道**：Twitter / Medium / Ghost
> **来源**：基于 Social Distribution Skill v1.1 审核流程改造
> **状态**：结构建议稿（待 CC 本地落地）

---

## 0. 顶层设计原则

### 0.1 与原 Skill 的差异

| 维度 | 原 Social Distribution v1.1 | 本 Skill |
|------|------------------------------|----------|
| 上游输入 | 已有运营素材 / 事件触发 | 全链路（含选题 + 内容生成 + 图） |
| 渠道 | X / TG / YouTube | Twitter / Medium / Ghost |
| 风险粒度 | 三级（🟢🟡🔴） | 短内容三级 + 长文二级 |
| Gate 数量 | 1 个（发布前） | 4 个（选题 / 草稿 / 配图 / 跨渠道） |
| 图片审核 | 不覆盖 | 占位（后期补，预留接口） |

### 0.2 核心设计原则

1. **Gate 串联,不并行**：选题不通过,后续生成不启动；草稿不通过,不进入跨渠道分发
2. **分级与 Gate 解耦**：风险等级决定每个 Gate 的"自动 / 人工"模式,Gate 本身永远存在
3. **长文与短文不混用同一套分级**：长文(Medium/Ghost)走二级模型,短文(Twitter)走三级模型
4. **图片 Gate 占位但不阻塞**：v1.0 仅做规则自检,不卡人工审核,等后期升级
5. **自动化降级有逃生通道**:任何 Gate 卡住超过 N 小时,触发"降级为草稿入库"而非永久阻塞

## 1. 双轨分级模型

### 1.1 短内容三级(Twitter)

```
🟢 低 → Agent 直接发布,事后抽检
🟡 中 → Agent 生成草稿 → 人工确认 → 发布
🔴 高 → 人工起草或全流程审核
```

**触发条件**：

| 等级 | 内容类型 |
|------|---------|
| 🟢 低 | 转推评论、节日问候、社区互动、教育型科普(无具体数据) |
| 🟡 中 | 含具体数据/指标、@合作方、产品功能介绍、行业观点 |
| 🔴 高 | 融资公告、合作公告、危机回应、法律/监管相关、价格相关 |

### 1.2 长文二级(Medium/Ghost)

```
📝 草稿态 → Agent 生成 → 人工审核 → 进入发布态
📤 发布态 → 人工确认 → 多渠道分发
```

**为什么长文不允许 🟢**:
- SEO 收录后难以擦除
- 外链引用形成长期影响
- 邮件订阅推送一次性,无法撤回
- 长文风险面 = Σ(每段事实风险),不是单点风险

**长文额外检查项**:
- 段落级事实校验(每个数据点单独标注来源)
- 引用合规(他人观点是否标注/链接)
- 免责声明模板是否就位(投资类/技术分析类必须)

### 1.3 升级触发器(自动从低级升高级)

任一条件命中,自动升级:
- 内容含币价/估值/收益数字
- @ 了未在白名单内的账号
- 提及未公开的合作方
- 涉及 SEC/合规/监管词汇
- 内容长度 >280 字符且含外链(Twitter Thread)

## 2. 四 Gate 流程

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ Gate A   │ → │ Gate B   │ → │ Gate C   │ → │ Gate D   │
│ 选题审核 │   │ 草稿审核 │   │ 配图审核 │   │ 跨渠道   │
└──────────┘   └──────────┘   └──────────┘   └──────────┘
   ↓决定           ↓决定           ↓决定           ↓决定
做不做         发不发         配不配         一致不
```

### 2.1 Gate A — 选题审核

**作用**:决定"这个话题值不值得做",卡在内容生成之前,避免做完才发现选题不行。

**输入**:话题候选(由热点抓取/事件触发/A-B-C 系列规划提供)

**自检项(Agent 自动跑)**:
- [ ] 选题与品牌定位匹配度(A-Builder/B-Workflow/C-Insights 哪一系列?)
- [ ] 是否与近 7 日已发内容重复
- [ ] 选题热度是否在窗口期内(>3 天热度衰减拒绝)
- [ ] 选题是否触碰红线词(监管/价格预测)
- [ ] 是否需要专业知识背书(超出已有知识储备直接拒绝)

**人工确认项(按分级)**:
- 🟢 短内容低风险 → 跳过人工,Agent 直接进 Gate B
- 🟡🔴 中高风险 / 长文 → 人工 1 分钟过一遍标题 + 大纲

**输出**:`topic_brief.md`(标题 / 系列归属 / 目标渠道 / 风险等级 / 关键事实点列表)

**卡控逻辑**:
- 通过 → 进入 Gate B
- 不通过 → 退回话题池,标记原因(避免重复推荐)
- 超时 24h 未审核 → 降级为草稿,等下次批量过

---

### 2.2 Gate B — 草稿审核

**作用**:发布前最后一道内容关,卡在配图和发布之前。

**输入**:`draft_content.md`(短/长内容草稿)

**自检 Checklist(Agent 必跑,继承自 v1.1)**:
- [ ] 事实准确性:所有数据/指标是否有来源标注?
- [ ] 合规性:是否涉及投资建议/收益承诺?(Web3 红线)
- [ ] 品牌一致性:调性是否符合产品定位?
- [ ] 链接有效性:所有 URL 是否可访问?(自动 HEAD 请求)
- [ ] 时效性:是否在最佳发布窗口内?(参考各平台数据)
- [ ] 跨渠道一致性:同一事件不同渠道口径是否统一?(留给 Gate D 深度检查,此处仅口径粗筛)
- [ ] 敏感词过滤:是否触发平台审核词汇?

**新增长文专用检查**:
- [ ] 段落级事实溯源:每个数据点是否有出处链接?
- [ ] 免责声明就位:涉投资/预测内容是否有 Disclaimer?
- [ ] 引用合规:他人观点是否标注作者+链接?
- [ ] SEO 元数据:Meta description / OG image / canonical URL 是否设置?

**人工模式**:
- 🟢 短内容 → 自检通过即放行
- 🟡 中风险 → 人工读全文 + 确认数据来源
- 🔴 高风险 → 人工起草 / 法律合规过审
- 📝 长文 → 强制人工通读

**卡控逻辑**:
- 通过 → 进入 Gate C
- 不通过 → 退回到内容生成节点,Agent 重写(最多 2 轮)
- 2 轮仍不通过 → 升级为 🔴 强制人工起草

---

### 2.3 Gate C — 配图审核(占位,v1.0 仅自检)

**作用**:防止配图引发版权/品牌一致性问题。

**v1.0 自检项(纯规则,不卡人工)**:
- [ ] 配图来源标注(自生成 / 授权图库 / 截图)
- [ ] 自生成图片 EXIF 是否注入水印
- [ ] 是否含可识别人脸(自动检测,触发即升级)
- [ ] 图片尺寸是否符合各渠道规范
  - Twitter: 16:9 / 1:1
  - Medium: 1500×750
  - Ghost: 1200×630(OG)

**v1.0 不做**:
- 真实人物肖像合规审核(后期补)
- 竞品视觉元素识别(后期补)
- A/B/C 系列视觉调性比对(后期补)

**预留接口**:`image_review_hook.md`(后期升级时直接替换该模块,不影响其他 Gate)

**卡控逻辑**:
- 自检命中"含人脸"或"无授权来源"→ 自动升级为🔴,转人工
- 其他情况 → 直接通过

---

### 2.4 Gate D — 跨渠道一致性审核

**作用**:多渠道分发时,确保同一事件在 Twitter/Medium/Ghost 的口径一致。

**仅在多渠道场景启用**(单渠道直接跳过)。

**自检项**:
- [ ] 核心数据点是否一致(数字/日期/链接)
- [ ] 关键术语是否统一(同一概念在不同渠道用同一表述)
- [ ] CTA 指向是否一致(都引导到同一着陆页/同一行动)
- [ ] 风险免责声明是否同步覆盖

**跨渠道发布顺序**(继承 v1.1 的"先发渠道反馈 hold 后续渠道"):
```
Step 1: 先发 Twitter(成本最低,反馈最快)
Step 2: 观察 30min,无负面反馈
Step 3: 发 Medium(中等成本,中等覆盖)
Step 4: 观察 2h,无投诉/纠错
Step 5: 发 Ghost(订阅推送,最不可逆)
```

**任一步暴露问题** → 暂停后续渠道,触发回滚流程。

**回滚静默期**(继承 v1.1):
- 低严重 → 立即更正,公开说明
- 中严重 → 静默 15-30min,修正后重发
- 高严重 → 沉默,不主动二次曝光

## 3. 快速通道(继承自 v1.1)

**触发条件**:被动事件评分 ≥7 且时效 <4h
- 跳过 Gate A 多版本生成
- Gate B 走 L1 快速审核(只看红线词 + 事实准确性)
- Gate C 强制使用预设视觉模板(不生成)
- Gate D 仅 Twitter 单发,不走多渠道

**目的**:避免热点过期,但代价是只能发短内容(Twitter),不允许走长文渠道。

## 6. 关键决策记录

**为什么 Gate C 占位而不删掉?**
- 删掉 → 后期补时整个 Gate 链条要重排,流程图作废
- 占位 → 仅替换该模块文件,SKILL.md 主流程不动

**为什么长文不允许 🟢?**
- 长文风险是累加性的(SEO + 订阅推送 + 外链),与短内容风险性质不同,无法用"事后抽检"兜底

**为什么 Gate D 仅多渠道启用?**
- 单渠道发不存在"一致性"问题,强行跑 Gate D 会拖慢自动化流速

**为什么快速通道禁止长文?**
- 快速通道的核心假设是"时效优先,可容忍小瑕疵",这个假设在长文场景不成立(长文修复成本太高)

## 7. 未决问题(留给本地 CC 讨论)

1. **Gate B 的"2 轮重写仍不通过 → 强制人工起草"** — 阈值 2 是凭直觉,需要跑一段时间收集数据再调
2. **跨渠道发布顺序"Twitter→Medium→Ghost"** — 这是按"反馈速度+不可逆性"排的,如果 Mobius 实际优先级是"Ghost 是核心订阅源,必须先发",顺序需要反转
3. **快速通道的评分体系(≥7 / <4h)** — 直接继承 v1.1,可能需要按新场景重新校准
4. **图片审核 v2.0 的人脸识别工具链** — 需要本地调研可用方案(Rekognition / Face++ / 本地模型)
